Python — 왜 데이터 분석의 표준 언어인가?

  • 인터프리터 언어: 소스코드를 한 줄씩 읽어 즉시 실행. 대화형(REPL) 탐색이 가능해 데이터 분석에 최적

  • 읽기 쉬운 문법: 들여쓰기 기반 블록 구조, 분석 코드를 비전공자도 읽을 수 있을 만큼 간결

Python 실행 구조 — Python은 인터프리터 언어다

파이썬은 컴파일 언어가 아닌 인터프리터 언어다. 사용자가 .py 파일로 작성한 코드를 한 줄씩 읽고 바로 실행한다. 그러나 단순한 한 줄씩 실행 구조가 아니라, 중간에 바이트코드(Bytecode)라는 중간 표현 단계가 존재한다. 인터프리터의 역할: 소스코드 → 파서(Parser) → AST(Abstract Syntax Tree) → 바이트코드 생성기 → Python Virtual Machine(PVM).

dis 모듈 — 바이트코드 확인
import dis
def add(x, y):
    return x + y
dis.dis(add)  # 바이트코드 출력

Python 실행 흐름 — 소스코드에서 결과까지

  • 소스코드(.py, 개발자 작성) → 파서(Parser, 문법분석) → AST(추상 구문 트리) → 바이트코드(.pyc, 중간표현) → PVM 실행(결과출력)
  • CPython은 소스를 직접 기계어로 변환하지 않고 바이트코드를 거친다 → 이식성과 재사용성 확보

[참고] AST(Abstract Syntax Tree) & dis

우리가 만든 코드를 기계가 실행할 수 있는 형태로 바꾸는 과정(컴파일), 코드에 문제가 없는지 분석하는 과정에서 AST가 핵심적인 역할을 한다.

  • 코드 분석 및 이해: 코드의 구조를 명확하게 보여줌 — 컴파일러·인터프리터가 코드 의미 분석 및 오류 검증
  • 코드 변환 및 최적화: 코드 구조를 트리 형태로 조작해 더 효율적인 코드로 최적화, 다른 버전의 언어로 변환
  • 다양한 개발도구 활용: 린터(Linter, 문법 오류 수정) · 포매터(Formatter, 스타일 통일) · 트랜스파일러(Transpiler, 최신 문법 변환)
  • dis 모듈은 함수가 어떤 바이트코드로 컴파일되는지 출력한다 — LOAD_FAST·BINARY_OP·RETURN_VALUE 등의 명령어 구조로 컴프리헨션이 for 루프보다 빠른 이유를 바이트코드로 확인할 수 있다
ast.dump()로 구문 트리 확인
import ast, dis
tree = ast.parse('x = a + b')
print(ast.dump(tree, indent=2))
def add(x, y): return x + y

# LOAD_FAST 0 (x)
# LOAD_FAST 1 (y)
# BINARY_OP 0 (+)
# RETURN_VALUE

Python 바이트코드 & PVM

  • 확인 예제: python -m py_compile hello.py → __pycache__/hello.cpython-311.pyc 생성됨
  • 바이트코드는 파이썬 소스코드가 파싱 후 생성하는 중간표현(Intermediate Representation), pyc 파일에 저장되며 PVM에서 실행 — 실행 속도 향상(재사용 가능), OS와 독립적으로 동작
  • PVM 역할: 바이트코드를 하나씩 읽어 해석하고 실행하는 엔진, 일종의 스택 기반 가상 머신 — 각 명령은 스택 기반으로 동작하며 바이트코드는 opcode + 인자(argument) 구조

def parse(val: str | int | None) -> str:

from pydantic import BaseModel, ValidationError
import csv
class Record(BaseModel):
    date: str
    amount: float
    region: str
valid, errors = [], []
with open('sales.csv') as f:
    for i, row in enumerate(csv.DictReader(f)):
        try:
            valid.append(Record(**row))
        except ValidationError as e:
            errors.append({'row': i, 'error': str(e)})
print(f'유효: {len(valid)}건, 오류: {len(errors)}건')

import asyncio, httpx
async def fetch(client, url):
    try:
        r = await client.get(url, timeout=10)
        return r.json()
    except Exception as e:
        return {'error': str(e)}
async def fetch_all(urls):
    async with httpx.AsyncClient() as c:
        tasks = [fetch(c, u) for u in urls]
        return await asyncio.gather(*tasks, return_exceptions=True)
urls = [f'https://api.example.com/data/{i}' for i in range(100)]
results = asyncio.run(fetch_all(urls))

from concurrent.futures import ProcessPoolExecutor
import multiprocessing as mp
def process_chunk(chunk):
    return [transform(row) for row in chunk]
def split_chunks(data, n):
    size = len(data) // n
    return [data[i*size:(i+1)*size] for i in range(n)]
n_cores = mp.cpu_count()
chunks = split_chunks(all_data, n_cores)
with ProcessPoolExecutor(max_workers=n_cores) as exe:

9. Polars + DuckDB

Lazy API · scan_csv · DuckDB SQL · Arrow · 도구 선택 기준

💡
쉽게 말하면: Pandas는 싱글스레드라 대용량에서 느려질 수 있어요. Polars는 멀티스레드로 훨씬 빠르고, DuckDB는 CSV/Parquet 파일에 로딩 없이 바로 SQL을 쓸 수 있어요. Lazy API는 "일단 계획만 세우고 collect() 부를 때 실제로 실행"하는 방식이라 더 똑똑하게 최적화돼요.

왜 Polars와 DuckDB인가 — 2026 표준

  • Pandas의 한계: 싱글스레드, 수백만 행 이상에서 속도 저하, 메모리 사용량이 원본 데이터의 5–10×, CoW 전까지 복사 많음
  • Polars — Rust + Arrow: 멀티스레드 자동 활용, Lazy API로 쿼리 최적화, Pandas 대비 5–20× 빠름, Apache Arrow 기반
  • DuckDB — 파일 직접 SQL: CSV·Parquet 파일을 로딩 없이 SQL 분석, 서버 없음, 로컬 DWH급 성능
  • Arrow 생태계 제로카피: Polars↔DuckDB↔PyArrow↔Pandas 간 Arrow 포맷으로 제로카피 변환, 데이터 이동 비용 최소

Polars — Eager API vs Lazy API

  • Eager(즉시 실행, 탐색·소규모에 적합), Lazy(scan_ 함수, 실행 계획 수립 후 collect())
  • Lazy 장점(predicate pushdown/projection pushdown), collect()(실제 실행 트리거), schema_overrides(컬럼 타입 명시, 성능↑)
Polars Lazy 체인 — scan_csv→filter→group_by→agg→collect
import polars as pl
df = pl.read_csv('sales.csv')
result = df.filter(pl.col('amount') > 0)
result = (
    pl.scan_csv('large.csv', schema_overrides={'amount': pl.Float64})
    .filter(pl.col('region') == '서울')
    .filter(pl.col('amount') > 0)
    .group_by('category')
    .agg([pl.col('amount').sum().alias('total'), pl.count().alias('cnt')])
    .sort('total', descending=True)
    .collect()
)

  • with_columns()(컬럼 추가·수정, Pandas: assign), group_by+agg(groupby+agg와 동일)

CRISP-DM — 실무 데이터 분석 방법론

업무 이해(문제 정의) → 데이터 이해(EDA 탐색) → 데이터 준비(전처리·피처) → 모델링(알고리즘) → 평가(성능 측정) → 배포(서비스 적용)

기술 중심이 아닌 문제 해결 중심 — 좋은 분석은 올바른 문제 정의에서 시작된다.

sklearn Pipeline — 전처리 + 모델 통합

  • fit()(학습)/predict()(예측)/transform()(변환), joblib.dump/load(파이프라인 직렬화)

from sklearn.preprocessing import StandardScaler, OneHotEncoder

import joblib

    ('cat', OneHotEncoder(), cat_cols)])

model.fit(X_train, y_train)
print(f'R2: {model.score(X_test, y_test):.3f}')

왜 분석을 자동화해야 하는가

  • 반복 분석의 현실: 매주 같은 매출 리포트를 수작업으로 만드는 분석가의 경우 자동화하면 수십 분 → 0분
  • 데이터 최신성: 어제 시황 및 뉴스, 산업 현황 수치 변화 등이 반영된 데이터로 만든 리포트는 오늘 오전 5시에 이미 완성돼 있어야 한다(증권가 애널리스트 사례)
  • 오류 감소: 수작업 복붙 오류 제거, 동일한 코드가 동일한 결과를 항상 생성 → 신뢰성
  • 확장성: 10개 지역 리포트를 만드나 100개를 만드나 코드는 동일, 규모 확장 비용 거의 없음

  • macOS launchd/cron으로 OS 레벨 스케줄, subprocess로 외부 스크립트 실행 연동

# src/clean.py
from typing import Optional
import pandas as pd
def clean_nulls(df: pd.DataFrame, cols: Optional[list] = None, strategy: str = 'median') -> pd.DataFrame:
    cols = cols or df.select_dtypes('number').columns.tolist()
    if strategy == 'median':
        return df.fillna(df[cols].median())
    elif strategy == 'drop':
        return df.dropna(subset=cols)
    return df

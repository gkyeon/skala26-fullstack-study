1. 데이터베이스 개요

필요성 · 발전 과정 · ACID · WAL · Isolation Level

💡
쉽게 말하면: 파일로 데이터 관리하다 한계(중복, 동시접속 문제)를 느껴서 DB가 나왔어요. ACID는 '계좌이체 같은 거래가 중간에 끊기거나 꼬이지 않게 보장하는 4가지 약속'. WAL은 '실제 잔고를 바꾸기 전에 통장 여백에 먼저 메모해두는 것'이라고 생각하면 쉬워요.

필요성과 발전 과정 (파일시스템 → 계층형 → RDBMS → 분산/클라우드)

  • 파일 시스템 시대: 앱마다 독자 파일 사용 → 중복 데이터, 일관성 어려움, 단일 사용자 중심
  • 계층형 DB(IBM IMS): 트리구조, 빠른 탐색이나 유연성 부족, N:M 표현 어려움
  • 관계형 DBMS(RDBMS): 1970년대 Codd 제안 → SQL 표준화, 인덱스·트랜잭션·정규화 등 최적화 기법 발전
  • 분산/클라우드 DB: 빅데이터·글로벌 서비스로 확장성/가용성 중시, NewSQL·NoSQL 혼합 (Spanner, Aurora, CockroachDB)

ACID 속성 & DBMS별 지원 차이

  • Atomicity(원자성) · Consistency(일관성) · Isolation(격리성) · Durability(지속성) — 트랜잭션이 지켜야 할 4대 원칙
  • ACID는 "교과서적으로는 모든 DBMS가 지원" — 실제 보장 수준은 엔진/설정마다 다름
  • MySQL InnoDB: 완전한 ACID (Redo/Undo Log, MVCC) / MyISAM: Rollback·FK 불가 → 사실상 ACID 미지원
  • PostgreSQL: MVCC + WAL + SSI, 모든 격리수준 지원 — 표준에 가장 가까움
  • Oracle: Read Committed 기본, 안정적 상용 구현 / Cloud DB(Aurora 등): 스토리지 6방향 복제로 장애 시 빠른 복구
WAL (Write-Ahead Logging) — 로그가 먼저
-- PostgreSQL WAL 동작 방식
BEGIN;
 UPDATE account SET balance = balance - 10000 WHERE id = 'A';
 -- 1단계: WAL 버퍼에 먼저 기록 (로그 우선 원칙)
 -- 2단계: Shared Buffer(메모리)에 페이지 변경 반영 (아직 디스크 미기록, dirty page)
 UPDATE account SET balance = balance + 10000 WHERE id = 'B';
COMMIT;
 -- COMMIT 로그를 WAL에 기록 후 fsync → 이후 Checkpoint 시점에 데이터 파일 반영
-- 장애 복구 시: pg_wal/ 로그를 순서대로 재실행(Redo), 미커밋 트랜잭션은 Undo
-- 설정: wal_level=replica / synchronous_commit=on(안전) / checkpoint_timeout=5min
WAL = 한 DB 안에서 장애 대비 안전장치 / Consensus(합의 알고리즘) = 분산 DB에서 여러 노드가 같은 결과를 갖도록 보장하는 안전장치 (네트워크 지연/장애 시에도)

Isolation Level — 동시성 이상 현상

  • Dirty Read: 커밋 전 값을 읽음
  • Non-Repeatable Read: 같은 행을 두 번 읽었는데 값이 다름 (다른 TX가 UPDATE/DELETE)
  • Phantom Read: 같은 조건 두 번 조회 시 행 개수가 다름 (다른 TX가 INSERT/DELETE)

-- Session 1
BEGIN;
SELECT COUNT(*) FROM orders WHERE amount >= 100; -- 결과: 5
-- Session 2 (동시)
BEGIN; INSERT INTO orders(id, amount) VALUES (999, 150); COMMIT;
-- 다시 Session 1

-- SERIALIZABLE: 팬텀 원천 차단 (범위/프레디킷 잠금 또는 SSI)

2. 관계형 모델 핵심 개념

테이블, 키(PK/FK/UK), 관계(1:1/1:N/N:M), 정규화(1NF~3NF)

💡
쉽게 말하면: 엑셀 시트 여러 개를 만들고 '학번' 같은 공통 열로 서로 연결(참조)해서 중복 없이 관리하는 방식이에요. PK=주민번호처럼 그 행을 딱 특정하는 값, FK=다른 시트를 가리키는 화살표라고 보면 됩니다.

테이블과 키 (PK / FK / UK)

  • 테이블(Table/Relation): 행(Row/Tuple)과 열(Column/Attribute)로 구성된 2차원 구조
  • 기본키(PK): 각 행을 고유 식별, NULL·중복 불가, 자동 인덱스 생성 (예: 학번)
  • 외래키(FK): 다른 테이블의 PK를 참조, 테이블 간 관계 표현·참조 무결성 보장
  • 고유키(UK): 중복 불가하지만 NULL은 허용 (예: 이메일, 주민번호)

관계 (Relationship)

  • 1:1 관계: 한 레코드가 다른 테이블의 레코드 하나와만 연결 (예: 학생 ↔ 학생증)
  • 1:N 관계: 가장 흔한 관계, 한 레코드가 여러 레코드와 연결 (예: 학생 1명 ↔ 성적 여러개)
  • N:M 관계: 중간(교차) 테이블로 구현 (예: 학생 ↔ 강좌 → '수강' 테이블)

정규화 (Normalization)

  • 정규화 목적: 데이터 중복 최소화, 이상현상(Anomaly) 방지, 무결성 향상
  • 삽입 이상: 불필요한 데이터도 함께 저장해야 함 / 삭제 이상: 삭제 시 다른 중요정보도 삭제됨 / 갱신 이상: 중복 데이터 일부만 수정 시 불일치
  • 1NF → 2NF(부분종속 제거) → 3NF(이행종속 제거) → BCNF, 보통 3NF 또는 BCNF까지 적용
  • 과도한 정규화 → JOIN 비용 증가 → 성능 고려 시 반정규화(Denormalization) 전략 도입
정규화 전/후 비교
-- 비정규화 (1NF 위반: 다중값 컬럼)
CREATE TABLE student_bad (
  student_id INT PRIMARY KEY, name VARCHAR(50),
  subjects VARCHAR(200) -- "DB, 네트워크, OS" ← 원자값 아님!
);
-- 1NF+2NF+3NF 적용 후 (분리된 테이블)
CREATE TABLE students ( id INT PRIMARY KEY, name VARCHAR(50) NOT NULL,
  major_id INT REFERENCES majors(id) ); -- 3NF: 학과명 분리
CREATE TABLE majors ( id INT PRIMARY KEY, code VARCHAR(20) UNIQUE, name VARCHAR(100) );
CREATE TABLE enrollments ( -- 1NF: 과목분리 + 2NF 교차테이블
  student_id INT REFERENCES students(id), course_id INT REFERENCES courses(id),
  score DECIMAL(5,2), PRIMARY KEY (student_id, course_id) );

3. ERD

Entity · Attribute · Relationship · ERD 작성법 · IE 표기법

💡
쉽게 말하면: 코드 짜기 전에 테이블 설계도를 그림으로 미리 그려보는 단계예요. 네모=테이블, 선=테이블 간 관계.

실습 환경

  • PostgreSQL(권장): 오픈소스, 표준 SQL 호환, JSONB/GIS/Vector 확장. 기본포트 5432(Mac 5433)
  • 실습 도구: dbdiagram.io(DBML 문법, SQL 내보내기), ERDCloud(한국어 UI), Lucidchart, DBeaver(GUI 클라이언트)

Entity

  • 엔티티: DB에서 관리할 대상 객체 (유형: 사람/사물, 무형: 사건/개념)
  • 강한 엔티티: 독자적 PK 존재, 혼자 식별 가능 (학생, 강좌)
  • 약한 엔티티: 독자적 PK 없음, 다른 강한 엔티티에 의존 (OrderItem = OrderID+ItemNo 복합키). ERD에서 이중 사각형, 식별관계는 굵은 선, 부모 삭제 시 자식도 삭제(ON DELETE CASCADE)

Attribute

  • 단순속성(더 이상 안 나뉨) vs 복합속성(주소=시+구+동)
  • 단일값속성(주민번호) vs 다중값속성(이메일 여러개 → 1NF에 따라 별도 테이블 분리)
  • 유도속성(나이=현재연도-출생연도, 직접 저장 불필요) / 기본키속성(ERD에서 밑줄로 표시)

IE 표기법 (까마귀발 표기법, Crow's Foot)

  • │ 하나(One) / 까마귀발 여럿(Many) / ○ 없거나 하나 / ∥ 반드시 하나
  • 식별관계(실선): FK가 자식의 PK 일부, 부모 없이 자식 존재 불가(약한 엔티티)
  • 비식별관계(점선): FK가 자식의 일반 컬럼, 부모 없이도 자식 존재 가능
dbdiagram.io DBML 예시
// dbdiagram.io (DBML) — SQL 없이 빠르게 ERD 작성 → SQL DDL 자동 생성
Table students {
  id bigint [pk, increment]
  student_no varchar(20) [unique, not null]
  major_id int [ref: > majors.id]
}
Table enrollments {           // N:M 교차 테이블
  student_id bigint [ref: > students.id]
  course_id int [ref: > courses.id]
  Indexes { (student_id, course_id) [pk] }  // 복합 PK
}
// 관계 방향: > 다대일, < 일대다, - 일대일, <> 다대다

ERD 작성 단계

  • 요구사항 수집 → 엔티티 도출(명사 추출) → 속성 정의 → 관계 설정(카디널리티 명시) → ERD 도구 시각화 → 검토 및 수정

5. 스키마 전략 & RDBMS 비교

멀티 프로젝트 설계 · MySQL · PostgreSQL · Oracle · Cloud DB

💡
쉽게 말하면: MySQL·PostgreSQL·Oracle·SQL Server는 각자 장단점 있는 자동차 브랜드 같은 거예요. 이 교재는 PostgreSQL 기준으로 진행됩니다.

멀티테넌시 3가지 패턴 (스키마 기반 멀티 프로젝트 설계)

  • Silo Model: 테넌트별 완전 독립 DB — 보안 최강, 운영비용 최대
  • Pool Model: 단일 DB + tenant_id 컬럼 — 운영 단순, 데이터 혼재
  • Bridge Model(PostgreSQL 권장): 단일 인스턴스 + 테넌트별 스키마 분리 — 물리적 자원 효율, 스키마 단위 접근제어/버전관리, 프로젝트별 독립 인덱스/파티셔닝 전략 가능

SELECT 논리적 실행 순서
SELECT s.student_no, s.name, m.name AS major_name,
       CASE s.grade WHEN 1 THEN '신입생' WHEN 4 THEN '졸업반' ELSE s.grade||'학년' END AS grade_label
FROM students s
  JOIN majors m ON m.id = s.major_id      -- 2. JOIN
WHERE s.enrolled = TRUE AND s.grade BETWEEN 1 AND 4   -- 3. 필터(집계 전)
ORDER BY s.grade DESC, s.name ASC
LIMIT 20 OFFSET 0;
-- 논리적 실행순서(작성순서와 다름!): FROM→JOIN→WHERE→GROUP BY→HAVING→SELECT→ORDER BY→LIMIT

10. SQL 중급 — 집계 & 그룹화

집계함수, GROUP BY, HAVING, ROLLUP, CUBE, FILTER

💡
쉽게 말하면: GROUP BY는 '학과별로 묶어서 평균 내줘' 같은 명령이에요. 엑셀 피벗테이블이랑 비슷하다고 생각하면 편해요.

집계 함수 핵심 5가지

  • COUNT(*): NULL포함 전체 / COUNT(col): NULL제외 / COUNT(DISTINCT col): 고유값 개수
  • SUM/AVG: NULL 자동 제외. AVG(NULL)=NULL (0이 아님에 주의!)
  • MIN/MAX: 문자열도 가능(알파벳/유니코드 순)
  • FILTER(PostgreSQL 전용): COUNT(*) FILTER (WHERE score>=80) — 다른 DBMS는 SUM(CASE WHEN...THEN 1 ELSE 0 END)
GROUP BY / HAVING
SELECT department_id, COUNT(*) AS emp_count, ROUND(AVG(salary),0) AS avg_salary
FROM employees
WHERE hire_date >= '2020-01-01'   -- 집계 전 행 필터 (WHERE)
GROUP BY department_id
HAVING COUNT(*) >= 3 AND AVG(salary) >= 5000   -- 집계 후 그룹 필터 (HAVING)
ORDER BY avg_salary DESC;
-- FILTER 절
SELECT COUNT(*) AS total,
 COUNT(*) FILTER (WHERE true_label = pred_label) AS correct,
 ROUND(COUNT(*) FILTER (WHERE true_label=pred_label)::NUMERIC / COUNT(*), 3) AS accuracy
FROM prediction_results;
ROLLUP / CUBE — 다차원 소계
-- ROLLUP: 계층적 소계 (지역별 → 전체합), 빠름
SELECT region, product, SUM(amount) AS total FROM sales_summary GROUP BY ROLLUP(region, product);
-- 결과: (East,A,100),(East,B,150),(East,NULL,250)... ,(NULL,NULL,500)
-- CUBE: 모든 조합 소계 (지역+상품+전체), 느림
SELECT region, product, SUM(amount) FROM sales_summary GROUP BY CUBE(region, product);
-- GROUPING 함수로 소계행에 라벨 부여
SELECT CASE WHEN GROUPING(region)=1 THEN '전체' ELSE region END AS region_label,
 SUM(amount) FROM sales_summary GROUP BY ROLLUP(region, product);

집계 쿼리 성능 최적화 포인트

  • SELECT에 GROUP BY 키 이외 컬럼 포함 금지 (MySQL ONLY_FULL_GROUP_BY 모드에서 에러)
  • OFFSET 페이지네이션 함정: LIMIT 10 OFFSET 990 → DB가 990개 읽고 버림 = O(N). 해결: Cursor/Keyset 방식 WHERE id > :last_id
  • ORDER BY + 인덱스: INDEX(dept_id, salary DESC)면 Sort 노드 제거 가능
  • 통계 테이블 전략: 자주 쓰는 집계값을 별도 테이블에 미리 계산(트리거 갱신) 또는 Materialized View + 주기적 REFRESH

11. SQL 고급 — JOIN 알고리즘

NL·Hash·Merge·BNL/BKA · DBMS별 엔진 비교

💡
쉽게 말하면: JOIN을 DB가 내부적으로 어떻게 처리하는지예요. 몰라도 쿼리는 짤 수 있지만, 알면 '왜 이 쿼리가 느린지' 이해하는 데 도움돼요.
알고리즘 동작 방식 적합한 경우
Nested Loop Join(NLJ)외부 테이블 행마다 내부 테이블 탐색외부 집합 작을 때 / 내부 인덱스 있을 때
Hash Join(HJ)작은쪽 해시테이블, 큰쪽 프로브탐색대용량 등가 조인, 인덱스 없을 때
Sort-Merge Join(SMJ)양쪽 정렬 후 병합 매칭이미 정렬된 대용량, 범위/비등가 조인
BNL/BKA(MySQL)여러 행을 버퍼에 모아 배치 조회join_buffer_size로 튜닝
실행계획으로 JOIN 알고리즘 확인
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT s.name, c.title, e.score FROM students s
 JOIN enrollments e ON e.student_id = s.id
 JOIN courses c ON c.id = e.course_id
WHERE s.grade = 3;
-- Hash Join (cost=12.5..45.8 rows=200) (actual time=0.5..2.3 rows=185)
-- Buffers: shared hit=12 read=5   ← hit=메모리(빠름), read=디스크(느림)
-- -> Seq Scan on enrollments  ← Full Scan, 인덱스 검토 필요
-- -> Index Scan on students   ← 인덱스 사용 중
-- MySQL 8.0.18+: Hash Batches > 1 → work_mem 부족 신호(Spill 발생)

JOIN 성능 튜닝 체크리스트

  • FK 컬럼에 반드시 인덱스! (ON DELETE CASCADE 대상 포함)
  • 작은×큰 테이블+인덱스 → Nested Loop / 대형×대형 → Hash
  • ON 조건 누락 주의 → 카르테시안 곱(N×M행) 발생
  • PostgreSQL: work_mem·hash_mem_multiplier 확인, Hash Batches>1 → Spill 신호
  • MySQL: join_buffer_size 설정, 8.0.18+ Hash Join 후보
  • SQL Server: tempdb Spill 감지 → 통계 갱신, Adaptive Join(2017+) 활용

12. JOIN 종류 상세

INNER · OUTER · SELF · CROSS · Anti-Join · Semi-Join

💡
쉽게 말하면: 여러 테이블을 합쳐 보는 여러 방법이에요. INNER=둘 다 있는 것만, LEFT=왼쪽 기준으로 다 보여주고 없으면 빈칸(NULL).
종류 설명
INNER JOIN두 테이블 모두 매칭되는 행만(교집합) — JOIN만 쓰면 기본값
LEFT OUTER JOIN왼쪽 전체 + 오른쪽 없으면 NULL
RIGHT OUTER JOIN오른쪽 전체 + 왼쪽 없으면 NULL (실무에선 LEFT로 방향 바꿔쓰는게 관례)
FULL OUTER JOIN양쪽 모두 보존. MySQL 미지원 → LEFT UNION RIGHT로 우회
SELF JOIN같은 테이블 2개처럼 조인 (직원-상사 계층)
CROSS JOIN카르테시안 곱(N×M) — 의도적으로만 사용
Anti-Join 패턴
-- Anti-Join: 한 번도 수강하지 않은 학생 (3가지 방법)
-- 1) LEFT JOIN + IS NULL (직관적)
SELECT s.name FROM student s LEFT JOIN enroll e ON e.student_id=s.id WHERE e.student_id IS NULL;
-- 2) NOT EXISTS (성능 우수, NULL 안전 - 권장!)
SELECT s.name FROM student s WHERE NOT EXISTS (SELECT 1 FROM enroll e WHERE e.student_id=s.id);
-- 3) NOT IN (NULL 함정 주의! 서브쿼리에 NULL 있으면 전체 결과가 빈 집합)
SELECT s.name FROM student s WHERE s.id NOT IN (SELECT student_id FROM enroll);

14. CTE · View · Materialized View

💡
쉽게 말하면: CTE(WITH절)는 복잡한 쿼리에 임시로 이름 붙여서 마치 테이블처럼 재사용하는 것, View는 자주 쓰는 쿼리를 아예 저장해두고 테이블처럼 불러쓰는 것이에요.
CTE(Common Table Expression) — WITH 절
WITH dept_avg AS (
  SELECT dept_id, AVG(salary) AS avg_sal FROM employees GROUP BY dept_id )
SELECT e.emp_name, e.salary, d.avg_sal, e.salary-d.avg_sal AS diff
FROM employees e JOIN dept_avg d ON d.dept_id=e.dept_id
WHERE e.salary > d.avg_sal ORDER BY diff DESC;
재귀 CTE — 계층형 데이터 탐색 (조직도, 카테고리 트리)
WITH RECURSIVE org_tree AS (
  -- 기저 단계(Anchor)
  SELECT id, emp_name, manager_id, 1 AS depth, emp_name::TEXT AS path
  FROM employees WHERE manager_id IS NULL
  UNION ALL
  -- 재귀 단계
  SELECT e.id, e.emp_name, e.manager_id, t.depth+1, t.path || ' > ' || e.emp_name
  FROM employees e JOIN org_tree t ON t.id = e.manager_id
  WHERE t.depth < 10   -- ← 무한재귀 방지! 항상 필요
)
SELECT depth, REPEAT(' ', depth-1) || emp_name AS indented_name, path FROM org_tree ORDER BY path;

15. Windows Function – 분석 함수

ROW_NUMBER · RANK · DENSE_RANK · NTILE · LAG · LEAD · 누적합 · 이동평균 (+부록: LATERAL, UPSERT, JSONB, RAG 등)

💡
쉽게 말하면: GROUP BY랑 비슷한데 그룹으로 묶어도 원래 행이 사라지지 않는 것이 핵심 차이예요. 순위 매기기(RANK), 전월 대비(LAG) 같은 데 씁니다. 이 세션은 분량이 많은데 뒤쪽 '[참고]'는 실무 심화 부록이라 처음엔 앞부분(순위/LAG/누적합)만 확실히 이해해도 충분해요.
GROUP BY Window Function
동작그룹으로 집계 → 행 수 줄어듦(개별행 소실)집계하면서도 원래 행 유지 → 행별결과+집계값 동시
기본 문법 — PARTITION BY(분석단위구분) · ORDER BY(윈도우내정렬) · ROWS/RANGE(집계범위 프레임)
함수명() OVER ([PARTITION BY 컬럼] [ORDER BY 컬럼] [ROWS/RANGE 프레임])

핵심 함수 분류

  • 순위함수: ROW_NUMBER(고유번호 1,2,3,4) / RANK(동점시 같은순위,건너뜀 1,2,2,4) / DENSE_RANK(동점시 같은순위,연속 1,2,2,3) / NTILE(n)(n개그룹균등분할)
  • 이동/비교함수: LAG(col,n) / LEAD(col,n) / FIRST_VALUE / LAST_VALUE
  • 집계Window: SUM/AVG/COUNT/MIN/MAX OVER() — 누적합, 이동평균
ROW_NUMBER / RANK / DENSE_RANK 비교
SELECT region, month, revenue,
 ROW_NUMBER() OVER (PARTITION BY region ORDER BY revenue DESC) AS row_num,
 RANK()       OVER (PARTITION BY region ORDER BY revenue DESC) AS rank_r,
 DENSE_RANK() OVER (PARTITION BY region ORDER BY revenue DESC) AS d_rank,
 NTILE(3)     OVER (PARTITION BY region ORDER BY revenue DESC) AS tier
FROM sales;
LAG/LEAD — 전월 대비 증감 (NULLIF로 0나누기 방지)
SELECT region, month, revenue,
 LAG(revenue,1) OVER (PARTITION BY region ORDER BY month) AS prev_month_sales,
 ROUND((revenue - LAG(revenue,1) OVER (PARTITION BY region ORDER BY month))
   / NULLIF(LAG(revenue,1) OVER (PARTITION BY region ORDER BY month),0) * 100, 1) AS mom_growth_pct,
 LEAD(revenue,1) OVER (PARTITION BY region ORDER BY month) AS next_month_sales
FROM sales;
-- 활용: MoM/YoY(LAG 12개월전) 성장률, 연속 이벤트 감지, 대기시간 계산
누적합 / 이동평균 — ROWS vs RANGE 프레임
-- 누적합
SUM(revenue) OVER (PARTITION BY region ORDER BY month
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cumulative_revenue
-- 3개월 이동평균
AVG(revenue) OVER (PARTITION BY region ORDER BY month
  ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) AS ma_3month
-- ROWS: 물리적 행 개수 기준(정확) / RANGE: 논리적 값(ORDER BY 값 같으면 모두 포함) 기준
-- 프레임 키워드: UNBOUNDED PRECEDING / N PRECEDING / CURRENT ROW / N FOLLOWING / UNBOUNDED FOLLOWING
실무 패턴 — TOP-N per Group / LATERAL JOIN / Gap&Island
-- TOP-N per Group (학과별 성적 상위 3명) — 3가지 방법
-- 1) Window Function (가장 추천)
SELECT * FROM ( SELECT s.name, e.score,
  ROW_NUMBER() OVER (PARTITION BY s.major_id ORDER BY e.score DESC) AS rn
  FROM students s JOIN enrollments e ON e.student_id=s.id ) ranked WHERE rn<=3;
-- 2) LATERAL JOIN (PostgreSQL, 행별 독립 서브쿼리)
SELECT m.name, top3.name, top3.score FROM majors m
CROSS JOIN LATERAL ( SELECT s.name, e.score FROM students s JOIN enrollments e ON e.student_id=s.id
  WHERE s.major_id=m.id ORDER BY e.score DESC LIMIT 3 ) top3;
-- Gap & Island: 연속 구간 찾기 (연속 로그인 등)
SELECT MIN(login_date), MAX(login_date), COUNT(*) FROM (
  SELECT login_date, login_date - ROW_NUMBER() OVER (ORDER BY login_date)*INTERVAL '1 day' AS grp
  FROM user_logins ) t GROUP BY grp;

17. 인덱스 설계 및 최적화

B-Tree · Hash · GIN · GiST · BRIN · 복합 인덱스 · 커버링 인덱스

💡
쉽게 말하면: 책의 '찾아보기(색인)'와 완전히 같은 개념이에요. 인덱스가 없으면 처음부터 끝까지 다 뒤지고(Full Scan), 있으면 바로 찾아갑니다.

인덱스 개념 및 트레이드오프

  • 인덱스 없음: Table Full Scan = O(N) / 인덱스: B-Tree 탐색 O(log N) — 수백만 행에서 수십ms 차이
  • 트레이드오프: 읽기 성능↑ vs 쓰기 비용↑(INSERT/UPDATE/DELETE시 재정렬) vs 저장공간 증가
  • 설계원칙: WHERE 자주 등장 컬럼 우선, 선택도(Selectivity) 높은 컬럼, FK 컬럼 필수, 미사용 인덱스 제거
B-Tree 인덱스 성능 비교 실습
-- Before: 인덱스 없이 (100만건)
EXPLAIN ANALYZE SELECT * FROM query_logs WHERE user_id='u123';
-- Seq Scan (actual time=120..1850)  ← 1850ms
CREATE INDEX idx_query_logs_user ON query_logs(user_id);
-- After: 인덱스 적용 후
-- Index Scan (actual time=0.05..0.8) ← 0.8ms (약 2,300배 향상)
인덱스 생성 명령어 모음
-- 복합 인덱스: 선두 컬럼 원칙(leftmost prefix rule)
CREATE INDEX idx_orders_cust_date ON orders (customer_id, order_date DESC);
-- 커버링 인덱스: SELECT 컬럼이 모두 인덱스에 있으면 테이블접근 불필요 → Index Only Scan
CREATE INDEX idx_customer_cover ON customers (customer_id) INCLUDE (name, email);
-- 부분 인덱스(Partial Index): 특정 조건 행만 인덱싱 → 크기↓속도↑
CREATE INDEX idx_orders_pending ON orders (created_at) WHERE status='PENDING';
-- 함수 기반 인덱스
CREATE INDEX idx_users_email_lower ON users (LOWER(email));
-- CONCURRENTLY: 운영 중 잠금 없이 생성
CREATE INDEX CONCURRENTLY idx_products_category ON products (category_id);

  • 복합 인덱스 설계: 등호(=) 조건 컬럼 먼저 → 범위(<,>) 조건 나중, 선택도 높은 컬럼 우선

  • 선두 컬럼 없이 order_date만 조건이면 사용 불가

인덱스 심화 — B-Tree 내부구조 & 커버링 인덱스

  • B-Tree = Balanced Tree: 모든 리프노드가 같은 깊이 → O(log N) 탐색 보장
  • 키가 작을수록 노드당 항목 많음 → 트리 깊이 감소 → 빠름 (UUID 16바이트 vs BIGINT 8바이트 → UUID가 더 큰 트리)
  • Fill Factor(기본 90%): UPDATE 많은 테이블은 낮추면(예:70%) 페이지 분할(Split) 감소
  • Index Only Scan: 커버링 인덱스로 Heap Fetches=0 달성 → 테이블 접근 없이 인덱스만으로 응답 (가장 빠름)

💡

실행계획을 읽어야 하는 상황

  • 쿼리가 갑자기 느려졌을 때 / 인덱스 만들었는데 안 쓰이는 것 같을 때
  • JOIN 방식이 이상할 때(Hash vs Nested Loop) / rows 예측값과 actual rows 크게 다를 때(통계 갱신 필요)
  • 시각화 도구: explain.dalibo.com (PostgreSQL JSON 플랜 붙여넣기)
PostgreSQL EXPLAIN ANALYZE — 플랜 읽기
BEGIN;
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT s.name, c.title, e.score FROM students s
 JOIN enrollments e ON e.student_id=s.id JOIN courses c ON c.id=e.course_id
WHERE s.major_id=1;
ROLLBACK; -- DML이면 롤백
-- Hash Join (cost=12.5..45.8 rows=200) (actual time=0.5..2.3 rows=185)
-- Buffers: shared hit=12 read=5  (hit=메모리, read=디스크)
-- Seq Scan on enrollments ← 인덱스 없음! 추가 검토
-- Planning Time: 0.2ms / Execution Time: 2.5ms

  • 예측 rows와 실제 rows 차이 크면 → ANALYZE 필요

타 DBMS 실행계획 확인

20. MVCC & 트랜잭션 격리 수준

다중버전 동시성 제어 · Isolation Level · DBMS별 실습

💡
쉽게 말하면: 여러 사람이 동시에 DB를 봐도 서로 안 꼬이게 하는 방법이에요. 핵심은 '수정해도 옛날 버전을 잠깐 남겨둔다(다중버전)'는 것.

MVCC(Multi-Version Concurrency Control) 심화

  • 데이터 수정 시 In-place 덮어쓰기 대신 새 버전 레코드 생성. 각 트랜잭션은 시작시점 스냅샷 보유 → 읽기가 쓰기를 차단하지 않음
  • PostgreSQL: 각 행에 xmin/xmax 시스템컬럼, UPDATE=새버전 INSERT+구버전 xmax 갱신. Dead tuple은 VACUUM이 회수. 장수 트랜잭션은 VACUUM 차단→Table Bloat 주의
  • MySQL InnoDB: Undo Log(변경전 데이터 별도 저장), Read View(TX시작시 스냅샷), Purge Thread(불필요 undo 자동정리)
Isolation Level 실습 — RC vs RR vs Serializable
-- Read Committed(기본): 커밋된 최신 데이터 읽음 → Non-Repeatable Read 발생 가능
BEGIN ISOLATION LEVEL READ COMMITTED;
SELECT balance FROM bank WHERE name='Alice'; -- 1000
-- (다른 세션이 800으로 UPDATE+COMMIT)
SELECT balance FROM bank WHERE name='Alice'; -- 800 (바뀜!)
-- Repeatable Read: 트랜잭션 내 항상 같은 값(스냅샷 유지)
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT balance ...; -- 1000
-- (다른 세션 UPDATE+COMMIT 해도)
SELECT balance ...; -- 여전히 1000
-- Serializable: 가장 엄격, 충돌 시 강제 실패
-- ERROR: could not serialize access due to concurrent update (에러코드 40001, 앱에서 재시도 필요)

21. Lock & Deadlock 관리

Row Lock · Advisory Lock · 감지 · 해소 · 모니터링

💡
쉽게 말하면: 두 작업이 서로 상대방이 가진 걸 기다리다 영원히 멈추는 상황(교착상태, Deadlock)과 그 해결법이에요.

Lock 종류 및 PostgreSQL Lock 구조

  • Row-Level Lock(Tuple Lock): 행단위 충돌방지(UPDATE/DELETE) / Table-Level Lock: ACCESS SHARE~ACCESS EXCLUSIVE / Advisory Lock: 사용자정의 논리락
  • FOR UPDATE(배타적 락, 대기) / FOR UPDATE NOWAIT(즉시 ERROR) / FOR UPDATE SKIP LOCKED(잠긴행 건너뜀 — 작업큐 처리에 유용)
SKIP LOCKED 패턴 (분산 작업큐/배치처리 필수 패턴)
-- SKIP LOCKED: 여러 Worker가 중복없이 작업 큐 분배 (경쟁 소비자 패턴)
WITH picked AS (
  SELECT id FROM job_queue WHERE status='READY' ORDER BY id
  FOR UPDATE SKIP LOCKED LIMIT 10 )
UPDATE job_queue SET status='RUNNING' FROM picked WHERE job_queue.id=picked.id
RETURNING job_queue.*;
Deadlock 발생 구조와 해소법
-- Deadlock: 교차대기 (Session1: row1잠금→row2요청 / Session2: row2잠금→row1요청)
-- → 순환대기! PostgreSQL이 자동감지 → ERROR: deadlock detected → 한쪽 자동 ROLLBACK
-- 해결1: 항상 동일 순서로 Lock 획득(ORDER BY id) — 핵심!
WITH targets AS (SELECT id FROM items WHERE id=ANY(ARRAY[1,2]) ORDER BY id FOR UPDATE)
UPDATE items SET stock=stock-1 FROM targets WHERE items.id=targets.id;
-- 해결2: 트랜잭션 짧게 유지 (HTTP호출/파일I/O는 트랜잭션 밖에서)
-- 해결3: Advisory Lock으로 업무단위 직렬화: SELECT pg_advisory_xact_lock(42);
-- 해결4: 재시도 전략(에러코드 40P01) — 지수 백오프 + Jitter
Deadlock 모니터링
-- 대기 중인 세션 확인 (blocking/waiting)
SELECT bl.pid AS waiting_pid, wl.pid AS blocking_pid, bl.query AS waiting_query
FROM pg_catalog.pg_locks l1 JOIN pg_catalog.pg_stat_activity bl ON bl.pid=l1.pid
JOIN pg_catalog.pg_locks l2 ON l1.locktype=l2.locktype AND l1.pid<>l2.pid
JOIN pg_catalog.pg_stat_activity wl ON wl.pid=l2.pid WHERE NOT l1.granted AND l2.granted;
-- 세션 강제 종료
SELECT pg_cancel_backend(pid);    -- 쿼리만 취소(graceful)
SELECT pg_terminate_backend(pid); -- 연결 자체 종료(강제)

22. 고급 DB 설계

BCNF·4NF·5NF · 반정규화 · SCD · 샤딩 · MSA DB 패턴 (+부록: 백업/VACUUM/모니터링)

💡
쉽게 말하면: 정규화를 더 깐깐하게(BCNF) 하거나, 반대로 성능을 위해 일부러 중복을 허용(반정규화)하거나, 데이터가 너무 많을 때 여러 서버로 쪼개는 것(샤딩)까지 다루는 심화 설계예요.

BCNF (Boyce-Codd Normal Form)

  • 조건: 모든 결정자(Determinant)가 후보키(Candidate Key)여야 함
  • 예: 학습지원(교수,과목,강의실) — 교수→강의실 함수종속(교수는 항상 같은 강의실) → 교수는 후보키 아닌데 강의실을 결정 = BCNF 위반
  • 해결: 교수강의실(교수,강의실) + 강의배정(교수,과목) 두 테이블로 분리
  • 3NF: 함수종속 보존 우선(비후보 결정자 허용 가능) vs BCNF: 정합성 우선(엄격). 실무는 3NF까지 적용(JOIN비용·복잡도 고려)

정규화 vs 반정규화 선택

정규화 유지(OLTP) 반정규화 허용(OLAP)
중심쓰기(INSERT/UPDATE) 중심, 무결성 최우선읽기(집계 SELECT) 중심, JOIN비용 감소 우선
적합은행계좌, 주문처리, 사용자등록대시보드, 보고서, AI/ML 피처테이블
기법3NF~BCNF 적용, 성능은 인덱스로 해결중복컬럼/집계컬럼 추가, 안전대안=Materialized View
SCD(Slowly Changing Dimension) — 이력 데이터 관리
-- SCD Type 1: 덮어쓰기(이력없음) — UPDATE customers SET city='부산' WHERE id=42;
-- SCD Type 2: 이력보존(새 행 추가, 기간관리) — 실무 표준
CREATE TABLE customer_history (
  id BIGSERIAL PRIMARY KEY, customer_id INT, name TEXT, city TEXT,
  valid_from DATE NOT NULL, valid_to DATE, is_current BOOLEAN DEFAULT TRUE );
-- 변경 발생시: 기존행 종료 + 새행 삽입
UPDATE customer_history SET valid_to=CURRENT_DATE-1, is_current=FALSE
  WHERE customer_id=42 AND is_current=TRUE;
INSERT INTO customer_history (customer_id,name,city,valid_from) VALUES (42,'홍길동','부산',CURRENT_DATE);
-- 특정 시점 조회: WHERE valid_from<=:date AND (valid_to IS NULL OR valid_to>=:date)
✔ Checkpoint
  • SCD Type 3: 현재+이전값을 별도 컬럼으로 보존
  • SCD Type 6: Type 1+2+3 혼합

샤딩(Sharding) — 수평 분할 전략

  • 파티셔닝: 단일 DB 인스턴스 내 논리/물리 분할 vs 샤딩: 여러 DB 인스턴스에 물리 분산(수평확장)
  • Range 샤딩(ID범위기준), Hash 샤딩(user_id % N, 균등분산이나 이관 어려움), Directory 샤딩(샤드맵 테이블)
  • 구현도구: Citus(PostgreSQL), Vitess(MySQL, YouTube 출신), TiDB(Spanner 영감)
  • 주의사항: Cross-shard JOIN 불가(앱레벨 처리), 분산 트랜잭션 복잡(2PC 필요), 샤딩 전 파티셔닝/Read Replica로 충분한지 먼저 검토
MSA DB 패턴 설명
Database per Service서비스별 독립 DB 소유 → 독립배포, 기술선택 자유. 단점: 분산TX 복잡, 데이터 일관성 어려움

PostgreSQL SP 실전 예 — 주문 생성 (재고부족 시 자동 ROLLBACK)
CREATE OR REPLACE PROCEDURE sp_create_order(p_customer_id BIGINT, p_items JSONB)
LANGUAGE plpgsql AS $$
DECLARE v_order_id BIGINT; v_item JSONB; v_stock INT;
BEGIN
  INSERT INTO orders (customer_id, status) VALUES (p_customer_id,'created') RETURNING id INTO v_order_id;
  FOR v_item IN SELECT * FROM jsonb_array_elements(p_items) LOOP
    SELECT stock_qty INTO v_stock FROM products WHERE id=(v_item->>'product_id')::INT FOR UPDATE; -- 동시성 처리
    IF v_stock < (v_item->>'qty')::INT THEN
      RAISE EXCEPTION '재고 부족: product_id=%', (v_item->>'product_id');  -- EXCEPTION시 자동 ROLLBACK
    END IF;
    UPDATE products SET stock_qty = stock_qty - (v_item->>'qty')::INT WHERE id=(v_item->>'product_id')::INT;
  END LOOP;
  UPDATE orders SET status='paid' WHERE id=v_order_id;
END;$$;

-- Row-level Trigger
CREATE OR REPLACE FUNCTION trg_sales_ai() RETURNS trigger AS $$
BEGIN INSERT INTO audit_log(table_name,op,row_json) VALUES ('sales', TG_OP, to_jsonb(NEW)); RETURN NEW; END$$ LANGUAGE plpgsql;
CREATE TRIGGER sales_ai AFTER INSERT ON sales FOR EACH ROW EXECUTE FUNCTION trg_sales_ai();

DECLARE r RECORD; BEGIN FOR r IN SELECT * FROM pg_event_trigger_ddl_commands() LOOP
  INSERT INTO ddl_history(event_tag, object) VALUES (r.command_tag, r.object_identity); END LOOP; END$$ LANGUAGE plpgsql;

28. 데이터 웨어하우스 & 분석 DB

OLTP vs OLAP · 파티션 프루닝 · 벡터화 실행 · Redshift · BigQuery · Snowflake · ClickHouse

💡
쉽게 말하면: 우리가 지금까지 배운 DB(OLTP, 실시간 거래처리용)와 분석·보고서 전용 DB(OLAP)는 애초에 설계 목적이 다르다는 걸 짚고 가는 파트예요.

분석 DB 핵심 기술 3가지

  • 파티션 프루닝: 필요한 날짜/범위 파티션만 읽음 (BigQuery PARTITION BY DATE, Snowflake 마이크로파티션, ClickHouse PARTITION BY)
  • 컬럼 프루닝: SELECT한 컬럼만 읽음 — SELECT * 절대 금지! (스캔 바이트=과금 기준)
  • 벡터화 실행: 컬럼단위 벡터(1000개씩)로 SIMD 처리 — BigQuery/Snowflake/DuckDB/ClickHouse 모두 채택

스타 스키마 & DW 설계 원칙

  • 스타 스키마: 팩트테이블(거래·이벤트, append-only, 시간파티션) + 디멘전테이블(사용자/상품/시간, SCD Type2)이 별모양으로 둘러쌈
  • Surrogate Key(디멘전 대체키), 역할차원(Role-Playing: 날짜 하나로 order_date/ship_date 별칭), Junk Dimension, Conformed Dimension(여러 마트 공용)
  • 스노우플레이크 스키마: 디멘전을 다시 정규화(계층 자주 바뀔 때) / 3NF DW: 원천레이어는 3NF, 소비레이어(mart)는 스타
BigQuery 파티션·클러스터 + ClickHouse MergeTree
-- BigQuery: 파티션+클러스터
CREATE TABLE mart.fact_sales PARTITION BY DATE(event_ts) CLUSTER BY user_id, item_id AS
SELECT event_ts, user_id, item_id, qty, amount FROM stg.sales_cleaned;

CREATE TABLE fact_events (event_time DateTime, user_id UInt64, amount Decimal(12,2))
ENGINE=MergeTree PARTITION BY toDate(event_time) ORDER BY (user_id, event_time)
TTL event_time + INTERVAL 90 DAY;

# ORDER BY 등 컬럼명은 바인딩 불가 → 화이트리스트 검증
if order not in ('name','created_at'): order = 'created_at'

31. 백업·복구 & 고가용성

Full/Incremental/PITR · Replication · Failover · Cloud 백업

💡
쉽게 말하면: DB 서버가 갑자기 죽어도 데이터와 서비스를 지키는 방법들이에요 (백업, 복제, 자동 전환).

Multi-AZ 구조 (고가용성)

  • Multi-AZ: Primary(읽기/쓰기)+Standby(복제전용) 두 AZ에 동기복제. Failover 시 AWS가 자동으로 Standby 승격
  • 동기복제 → 쓰기완료엔 양쪽 모두 필요 → RPO(데이터손실)≈0 / RTO(복구시간)는 수 분 내
PostgreSQL 백업/PITR(Point-In-Time Recovery)
-- 논리백업
pg_dump -U postgres -d mydb -Fc -f backup.dump  -- 압축바이너리(권장), -j 4로 병렬가능
pg_dumpall -U postgres > full_cluster_backup.sql -- 전역객체 포함 전체
-- 물리백업(PITR용)
pg_basebackup -U replicator -D /backup/base -Ft -z -P --wal-method=stream
-- postgresql.conf: wal_level=replica, archive_mode=on, archive_command='cp %p /backup/wal/%f'
-- 복구 시점 지정: recovery_target_time = '2024-12-15 14:30:00'
백업 원칙: 3-2-1 (3 copies / 2 media / 1 offsite) + 정기 복원 테스트 필수!

Streaming Replication & DR 목표

구분 설명
Streaming ReplicationPrimary→Standby 실시간 WAL 복제, Hot Standby로 읽기분산
SynchronousCOMMIT시 Standby 응답 대기 (RPO=0, 성능저하)
AsynchronousCOMMIT 즉시 응답 (RPO>0, 기본값, 성능우수)
RPORecovery Point Objective — 최대 허용 데이터 손실 시간
RTORecovery Time Objective — 최대 허용 서비스 다운 시간

고가용성 솔루션 심화

  • Failover 도구: Patroni(Python HA매니저, 자동승격) / pg_auto_failover, Repmgr(대안)
  • Oracle RAC: 여러 인스턴스가 공유스토리지 동시접근, Cache Fusion, 대형엔터프라이즈
  • SQL Server Always On AG: DB집합단위 HA/DR, 동기(RPO=0)/비동기(DR) 선택
  • PostgreSQL FDW: postgres_fdw로 원격 테이블 로컬처럼 참조, 이기종DB조인/점진적마이그레이션

지표 의미 비유(카페)
